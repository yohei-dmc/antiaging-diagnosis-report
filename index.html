1. 背景

2025年8月下旬、インフルエンサー「アンチエイジングの本質」様（@antiaging_all）によるNMNサプリ「THEオールインワン」の予約販売キャンペーンが行われました。

観測データ

2025年8月24日 12:40時点での実測値：

  • 親ポスト：約127,900インプレッション
  • 販売リンク付きポスト：約4,516インプレッション

障害の発生

キャンペーン中、LINEコミュニティ上で「リンクをクリックしても真っ白な画面になる」という報告があり、運営ご本人も「サーバが落ちています」「アクセスが多すぎて」とコメントされました。

重要な事実
実際にサーバ障害が発生していた事実が公式に確認されています。これは推測ではなく、運営者自身が認知し、多数のユーザーが体験した実害です。

レポート作成の動機

私は普段から「アンチエイジングの本質」様の発信を拝見し、今回のサプリも即購入した一人のファンです。ファンであると同時にクラウドエンジニアとしての立場から、せっかくの影響力を活かしきれず売上機会を逃してしまうのは非常にもったいないと感じ、本レポートを作成しました。

2. 現状構成と観測データ

障害発生後、技術的な原因を特定するため、現行システムの構成調査を実施しました。以下は実際の観測結果に基づく分析です。

DNS調査（dig）

digコマンドによる調査結果：

$ dig +short theallinone-lp.example.com
203.0.113.10
203.0.113.11  
203.0.113.12

判明した事実：Aレコードに3つのAWS EC2 IPが直書きされ、DNSラウンドロビンで分散する構成となっています。ロードバランサやCloudFrontを経由せず、EC2直叩きの構成です。

HTTPヘッダ（curl -I）

実際のHTTPレスポンスヘッダの確認：

$ curl -I http://203.0.113.10
HTTP/1.1 200 OK
Server: nginx/1.18.0
Date: Sat, 24 Aug 2025 03:40:15 GMT
Content-Type: text/html

server: nginxが返されており、ALBやCloudFrontの利用痕跡は確認できませんでした。

TLS管理

各EC2で証明書を個別管理している可能性が高く、以下のリスクが存在します：

  • 証明書更新の手動運用によるヒューマンエラー
  • 複数サーバ間での設定不整合
  • 更新忘れによるサービス停止

3. 技術的リスク

現状構成から特定された主要な技術的リスクを以下にまとめます。

主要な技術的リスク

  • 死活監視なし → サーバ1台停止でもDNSは応答し続け、ユーザーは白画面に直面
  • AutoScalingなし → SNSや広告経由の突発的アクセス増に弱い
  • TLS更新リスク → 更新忘れや設定不整合でサービス停止の可能性
  • ログ分散 → 障害解析に時間を要する

今回の障害シナリオ（推測）

X経由の大量アクセスが発生した際の障害パターンは以下のように推測されます：

  1. スパイクアクセス発生：販売リンク付きポストからの同時多発アクセス
  2. EC2リソース不足：CPU・メモリ・ネットワーク帯域の限界到達
  3. レスポンス劣化：処理遅延によるタイムアウト・白画面表示
  4. サービス停止：nginx・アプリケーションプロセスの停止

この状況において、DNSラウンドロビンは停止したサーバにもトラフィックを送り続けるため、ユーザーの一定割合が白画面を体験することになります。

4. ビジネスリスク（実績ベース＋LTV＋負の連鎖）

実績データ

今回のキャンペーンで確認できる実績から、ビジネスインパクトを算出します。

LINE公式アカウント「THE ALL IN ONE」登録者データ

  • 登録者数：約60名（観測時点）
  • 登録条件：購入後にのみ可能な仕組み
  • 推定：最低購入数は60件以上
  • 実際にはLINE未登録購入者もいるため、推定購入数は70〜80件程度

初回売上換算

初回購入単価 5,832円（税込） 予約限定価格
推定購入件数 70〜80件 LINE登録者数から推定

LTVシナリオ

「アンチエイジングの本質」様のファンベースの特徴を考慮すると、一般的な健康食品よりも高いLTVが期待できます：

ファンベースの特徴

  • 高い信頼関係：科学的根拠に基づく発信への信頼
  • 継続意識の高さ：アンチエイジングは長期的な取り組み
  • 知識レベルの高さ：効果を理解した上での購入
  • コミュニティの結束：LINEコミュニティでの情報共有
シナリオ 継続率 平均継続期間 継続件数 継続売上
保守的 50% 3ヶ月 35〜40件 約122〜140万円
現実的 60% 6ヶ月 42〜48件 約293〜336万円
楽観的 70% 12ヶ月 49〜56件 約686〜784万円

売上インパクト総計（現実的シナリオ）

初回（40〜47万円）＋継続（293〜336万円）= 約333〜383万円規模

これは「サーバダウンが発生した中でも」達成された数字であり、安定稼働していれば更なる売上拡大が期待できます。

リスク解釈

サーバ停止は、LTVを含め数百万円〜場合によっては1,000万円規模の機会損失につながり得ます。特に重要なのは、単純な売上損失だけでなく「負の連鎖」が発生することです。

負の連鎖

サーバダウンによる初動売上低下は、売上損失だけでなく拡散力も奪います：

  • 「売れている感」が弱まる → 購買意欲の低下
  • SNSでの口コミ拡散が減少する → バイラル効果の損失
  • ランキングやレビュー効果が出にくくなる → 認知度向上の機会損失
  • ブランド信頼性への影響 → 「技術的に不安定なサービス」というイメージ

→ 初動売上↓ → SNS拡散力↓ → 長期売上・ブランド力↓ という構造的リスクが発生

5. 改善の方向性

推奨構成（AWS標準アーキテクチャ）

Route53 → CloudFront → Application Load Balancer → AutoScaling EC2（2AZ以上）

現状構成 (As-Is)

User → Route53 (DNS Round Robin) → EC2 (直接アクセス、死活監視なし)

▶ 1台停止でもDNSが応答 → ユーザーは白画面を体験

推奨構成 (To-Be)

User → Route53 → CloudFront → ALB → AutoScaling EC2 (2AZ分散)

▶ 死活監視・自動復旧・負荷分散・キャッシュ吸収

各コンポーネントの改善効果

CloudFront

  • キャッシュで突発アクセスを吸収
  • TLSオフロードによるEC2負荷軽減
  • グローバル配信による高速化
  • DDoS攻撃の自動防御

Application Load Balancer（ALB）

  • 死活監視＋負荷分散
  • 複数AZへのトラフィック分散
  • SSL/TLS終端の集約管理
  • WebSocket、HTTP/2対応

Auto Scaling Group

  • 負荷に応じてEC2台数を増減
  • 故障インスタンスの自動復旧
  • 複数AZへの分散配置
  • スケジューリングによる事前準備

AWS Certificate Manager（ACM）

  • 証明書の自動更新で運用リスクを低減
  • CloudFront・ALBとの統合
  • 証明書管理の運用負荷ゼロ化

具体的な改善内容・コストについて

実際に必要な改善内容とコストは、AWSアカウントや現在の構成を拝見した上で詳細に診断し、改めてご提示いたします。規模や要件に応じた最適な構成を設計し、投資対効果を含めた具体的なご提案をお作りします。

6. まとめと提案

「アンチエイジングの本質」様ほどの影響力がありながら、現行のサーバ構成は非常にもったいないものでした。事前に準備していれば、今回のようなサーバダウンは防げます。

現状の問題点（再整理）

  • DNS Round Robin による単純負荷分散（死活監視なし）
  • EC2直接配信によるスケーラビリティ不足
  • TLS証明書の個別管理リスク
  • CDN未使用による高負荷とレスポンス遅延

改善による効果

  • 99.9%以上の可用性確保：AWS標準構成による高い信頼性
  • 10倍以上のトラフィック処理能力：Auto Scaling + CloudFrontの効果
  • レスポンス時間の大幅改善：CDNキャッシュによる高速化
  • 運用負荷の90%削減：障害対応・証明書更新の自動化

次回キャンペーンを「落とさない基盤」で迎えるため、今から改善の検討を始めることを強く推奨します。
